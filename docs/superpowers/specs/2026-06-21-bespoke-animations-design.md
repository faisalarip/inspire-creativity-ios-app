## 3. How an animation exists today (baseline)

Each catalog animation is wired across four surfaces:

1. **Preview view** — a SwiftUI `View` in `Animations/Previews/*.swift`.
2. **Registry** — `AnimationPreviewRegistry.builders[id] = { AnyView(SomePreview()) }`.
3. **Metadata** — an `AnimationItem` in `AnimationCatalogSeed` (`handcrafted` or `auroraDerived`).
4. **Paste-ready source** — `swiftCode`, either a `CodeSamples.Code.xxx` literal (handcrafted) or generated by `AuroraCodeGen` (aurora).

Presentation surfaces (confirmed):
- **Grid tile** (`AnimationCard`, 116–160pt) → `AnimationPreviewRegistry.view(for:)` — auto-playing, non-interactive, paused off-screen via `previewsPaused`.
- **Detail** (`DetailView`, ~42% height) → same `view(for:)` today; drag is reserved for the code sheet.

Baseline build is **green** (`xcodebuild … -scheme InspireCreativityApp -destination 'platform=iOS Simulator,name=iPhone 17 Pro'`, Xcode 26.3, iOS 18+ APIs available).

## 4. Target architecture

### 4.1 The bespoke-animation contract (uniform across all 210)

Each animation lives in **one self-contained file**:

```
InspireCreativityApp/Animations/Catalog/<Category>/<TypeName>.swift
```

The file contains exactly one public component conforming to this shape:

```swift
import SwiftUI   // SwiftUI ONLY — no Theme/HexColor/app deps (paste-ready)

struct <TypeName>: View {
    /// `demo == true`  → self-driving loop for the grid tile (simulates the gesture, no finger).
    /// `demo == false` → real, finger-interactive component for Detail + the buyer's code.
    var demo: Bool = false
    var body: some View { … }
}
```

- **`demo: true`** drives itself (a `PhaseAnimator`/`TimelineView` cycles the gesture's progress 0→1→0) so the grid tile looks alive with no touch.
- **`demo: false`** wires the real gesture (`DragGesture`, `MagnifyGesture`, `RotationGesture`, …).
- The file is **self-contained and paste-ready** (embeds its own `Color(hex:)` etc. when needed), exactly like `AuroraCodeGen`'s output — so its source *is* the `swiftCode`.
- **Metal Shaders exception:** custom shaders need a companion `.metal` file in the target (SwiftUI can't compile a shader from a string). So a Metal item is *two* files — `XxxView.swift` (marked `// catalog-metal: Xxx.metal`) + `Xxx.metal`. The generated `swiftCode` embeds the `.metal` source as a trailing comment block so the snippet is still complete. Requires the Metal toolchain (see Gate 1 findings).

### 4.2 Registry extension (additive, zero churn to existing 107)

```swift
enum AnimationPreviewRegistry {
    static let builders: [String: PreviewBuilder] = [ … existing … ]      // grid (loop)
    static let interactiveBuilders: [String: PreviewBuilder] = [ … new … ] // detail (interactive)

    static func view(for id:)            // unchanged → grid, self-driving
    static func interactiveView(for id:) // NEW → detail; falls back to view(for:) when absent
}
```
New animations register `builders[id] = { AnyView(XxxView(demo: true)) }` and `interactiveBuilders[id] = { AnyView(XxxView(demo: false)) }`. Existing 107 have no interactive builder → `interactiveView` falls back to the loop. **`DetailView` switches its preview call to `interactiveView(for:)`.**

### 4.3 Metadata

New items go in `AnimationCatalogSeed.bespoke: [AnimationItem]` (split into per-category files under `Repositories/Seed/` to keep files focused), appended into `items`. Existing Pro→$10 normalization applies unchanged.

### 4.4 Paste-ready `swiftCode` — generated, never hand-duplicated

A generator reads each `<TypeName>.swift`, strips the registry plumbing, and emits a `BespokeCodeSamples.swift` mapping `id → source string`. Because the source *is* the component, **the copied code always matches the preview**. Regenerated on every change; no drift, no 210 hand-maintained literals.

### 4.5 pbxproj integration (deterministic)

A ruby script (`Tools/add_sources.rb`, `xcodeproj` gem) adds every new `.swift` to the `InspireCreativityApp` target's sources phase and the correct group. Idempotent; verified by a green build. No manual `project.pbxproj` edits.

## 5. Execution strategy (the build)

The serial bottleneck is the **build**, not the Swift. Success is decided by *sequencing and what we prove first*, so execution runs in three gates — never bulk-generate 210 before the pipeline is proven.

### Gate 1 — Vertical slice (hand-authored, not delegated)
Hand-author **3 animations** end-to-end through the *real* tooling: file → `add_sources.rb` (pbxproj) → registry (grid + interactive) → seed → `DetailView` switch → codegen → `xcodebuild` green → **launch sim and actually see it** (grid loop + Detail gesture). Hand-authoring means a tooling bug can't hide behind a generation bug. The slice deliberately covers the scariest paths:
- **1 Metal-shader** item (`mtl-…`),
- **1 iOS-18 `TextRenderer`** item (text effect, 18+),
- **1 heavy-physics vertical-drag gesture** (e.g. `ges-rubberband-sheet-morph`) — the user's hero example.

**Critical unknown the slice must resolve:** `DetailView` renders the preview **inside a `ScrollView`** (DetailView.swift:59–65). A vertical `DragGesture` in an interactive piece will fight the scroll. The slice determines the fix (`.simultaneousGesture`, disable-scroll-while-interacting, or a contained drag region) — and that answer reshapes the `interactiveSpec` contract for every vertical-drag category. Resolve here, not at batch 11.

### Gate 2 — Calibration batch (~18, measured)
Run **one** full batch (generate → adversarial review → integrate → build) and measure the **compile-pass rate**. Tune the generation prompt against real failures before spending tokens on the other ~190.

### Gate 3 — Bulk
Remaining batches of **~18** → integrate → `xcodebuild` green → **commit each** → repeat. One bad view can't hide behind 200 others.

### 5.1 Generation workflow shape
```
pipeline(concepts,
  concept → agent: author self-contained XxxView (demo flag) + seed entry  [schema]
  authored → agent: adversarial compile/quality review of that view        [schema, verdict]
)
→ collect verified → deterministic integration (files + registry + seed + pbxproj + codegen)
→ xcodebuild (main-loop, NOT in the workflow) → commit
```
Agents return **data, never file edits** (the 4 central files would conflict under parallel writes), so `isolation: 'worktree'` is not used. **The build is run by the main loop** — workflows only produce verified data; only integration + `xcodebuild` proves the target.

### 5.2 Generation/review landmines (baked into prompts)
- **SwiftUI type-checker timeouts** ("unable to type-check in reasonable time") — a hard failure in fat animated bodies. Generators must break up expressions, annotate types, and factor subviews; the reviewer flags fat bodies.
- **SwiftUI-only, zero app deps** — no `Theme`, `HexColor`, or design-system references. This is the entire basis of `swiftCode = source`; it's an explicit **reject** criterion. (Accepted consequence: catalog animations don't use app Theme colors — fine for a standalone catalog.)
- **Deterministic `typeName`** — derived centrally from `id`, uniqueness checked across all 210 **plus** the existing 107 symbols (`SpringButtonPreview`, etc.). Agents don't name their own structs (collisions surface only at link time).
- **Sanitized folder/group names** — `Micro-interactions`→`MicroInteractions`, `Text effects`→`TextEffects`, `Metal Shaders`→`MetalShaders`.
- **Demo loop must never go fully blank** — a `previewLoop` that drops to zero opacity / off-screen (e.g. a shatter scatter phase) leaves grid tiles momentarily empty. Keep a legible minimum state every frame.

### 5.3 Gate 1 — outcome (PASSED, on branch `feat/bespoke-animations-catalog`)
Hand-authored slice of 3 (`ges-rubberband-sheet-morph`, `tx-shatter-glass`, `mtl-heat-mirage`) proved the full pipeline: file → `add_sources.rb` → registry (grid+interactive) → `BespokeSeed` → `gen_codesamples.py` → `xcodebuild` **green** → all tests pass → app launches & renders the new tiles (correct Pro/iOS badges, no crash). Findings, all resolved:

1. **Metal toolchain required (Xcode 26).** `.metal` files fail with *"missing Metal Toolchain"* until `xcodebuild -downloadComponent MetalToolchain` is run once (≈705 MB, now installed). Without it, the Metal Shaders category can't build.
2. **`ScrollView` vs. vertical drag.** `DetailView`'s preview sits in a `ScrollView`; an interactive vertical drag was at risk of being stolen by scroll. **Resolved generically** in `DetailView`: a `simultaneousGesture(DragGesture(minimumDistance: 0))` flips `previewInteracting` at touch-down and `.scrollDisabled(previewInteracting)` frees the preview's own gesture. No per-category special-casing needed.
3. **Free-taster invariant.** `CatalogGatingTests` requires exactly 20 free items → all bespoke forced Pro (see §2 Pricing).
4. **Toolchain:** scheme `InspireCreativityApp`, sim `iPhone 17 Pro`, Xcode 26.3, bundle `com.inspirecreativity`. Catalog counter rose 109→112 confirming the merge.

## 6. Risks & mitigations

| Risk | Mitigation |
|---|---|
| 210 views = large compile surface; one error fails the target | Batch build (~18) → green → commit; never "generate all then build once". |
| pbxproj corruption | `xcodeproj` gem only; idempotent; build-verified; commit per green batch. |
| `swiftCode` drifting from preview | `swiftCode` is generated *from* the view source — they can't drift. |
| Gesture pieces look dead in the auto-play grid | Mandatory `demo:true` self-driving loop per `previewLoop` spec. |
| Off-screen GPU burn | New loops honor existing `previewsPaused` (wrapped by `PreviewStage`). |
| Concept duplication / clichés | Curated catalog (deduped vs existing 107) + per-batch dedup review. |
| iOS 18-only APIs (TextRenderer, MeshGradient, some symbol effects) | `iosVersion` set per item; `#available` guards; only 5 items are 18+. |

## 7. Out of scope

Backend/Supabase remote catalog rows, paywall/pricing changes, new categories, analytics. (Pricing reuses the flat-$10 Pro normalization already in `AnimationCatalogSeed`.)

## 8. Verification

- Per batch: `xcodebuild … build` green; existing `InspireCreativityAppTests` still pass (`BrowseFilterTests`, `CatalogPolicyTests`, etc.).
- Spot-run in the iPhone 17 Pro simulator: grid tiles animate; Detail responds to real gestures; copied code compiles standalone.
- **Browse GRID check (not just single tiles):** many `TimelineView(.animation)` loops on screen at once can cause GPU jank even with off-screen pausing. Confirm `previewsPaused` actually unmounts a `TimelineView` loop (`PreviewStage` swaps to `Color.black`).
- Final: full catalog count = 107 + 210 = 317; per-category counts match §2.

### Verification honesty (the ceiling)
- **Build-green ≠ looks-wonderful.** All 210 are *compile*-verified; only a sample is *visually* verified. This will be stated plainly in the final report — no implication that 210 were each eyeballed.
